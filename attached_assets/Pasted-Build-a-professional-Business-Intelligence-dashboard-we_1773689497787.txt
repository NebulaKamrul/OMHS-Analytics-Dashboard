Build a professional Business Intelligence dashboard web application using a real Ontario mental health services Excel dataset.

The goal is to simulate an internal analytics/reporting tool that policy analysts, administrators, or BI teams could use to understand the distribution, accessibility, and eligibility patterns of mental health services across Ontario.

This project is intended for a technical internship portfolio, so it should prioritize realism, structured data modeling, SQL-based analytics, clean enterprise UI, and strong engineering practices rather than flashy design.

The uploaded Excel file is the main data source. It contains mental health service records with fields such as public service name, official name, taxonomy/category, mailing and physical location fields, latitude/longitude, eligibility fields, languages offered, coverage area, hours, wait time, and custom fields such as bilingual service, eligibility by age, eligibility by gender, LGBTQ support, and harm reduction.

-----------------------------------
PROJECT GOALS

This project should demonstrate:

• cleaning and transforming a messy real-world Excel dataset
• relational data modeling
• SQL-based analytics queries
• dashboard reporting
• filterable reports
• data visualization
• enterprise-style UI discipline
• clear backend data flow

This should feel like a practical internal reporting dashboard used by analysts or service planners, not a startup landing page.

DATA INGESTION

STRICT DESIGN RULES

Avoid vibe-coded website patterns.

COLORS & VISUAL
• No default purple gradients unless brand-appropriate
• No sparkles or emojis in headings
• No glowing hover effects
• Use a neutral enterprise palette
• Maintain consistent contrast

TYPOGRAPHY
• Define a consistent type scale
• Avoid oversized headings with ultra-thin body text
• Maintain consistent line height
• Maintain consistent paragraph spacing
• Enforce clear hierarchy

LAYOUT & COMPONENTS
• Consistent component placement across pages
• Define and reuse 2-3 border radius values maximum
• Subtle hover states only (2-4px lift max)
• Icons scaled proportionally to text
• Remove non-functional social icons
• Avoid decorative UI elements with no functional purpose

ANIMATIONS & INTERACTIONS
• Use easing curves (cubic-bezier)
• Stagger timing intentionally
• Animations must serve a UX purpose
• Avoid decorative animation

UX BEHAVIORS
• Loading states for all async actions
• Progress indicators for long operations
• Skeleton screens for data-heavy sections
• Functional filters and interactions
